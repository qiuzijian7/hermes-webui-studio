# 测试执行技能

## 核心原则
**测试的目的是发现缺陷，不是证明无缺陷。带着"怎么搞坏它"的心态。**

## 测试金字塔

```
        /\
       /  \  E2E  (少量，核心流程)
      /----\
     / 集成  \  (中量，跨模块场景)
    /--------\
   / 单元测试 \  (大量，快速反馈)
  /____________\
```

## 探索性测试清单（发现 bug 的宝库）

### 边界值
- 输入 0 / 1 / 最大值 / 最大值 +1 / 最大值 ×2
- 空字符串 / 单字符 / 超长字符串 / Unicode / emoji
- 负数 / 小数 / NaN / Infinity

### 状态组合
- 首次打开 vs 第 100 次打开
- 有网络 vs 断网 vs 慢网 vs 网络抖动
- 空数据 vs 有数据 vs 数据超量
- 登录 vs 未登录 vs 登录过期

### 并发 / 时序
- 快速连点同一按钮
- 中途切换页面 / 关闭 / 回退
- 多设备同时操作同一账号
- 操作过程中系统时间变化（跨日 / 时区）

### 资源异常
- 磁盘满
- 内存压力
- CPU 100%
- 电量低

## Bug 报告模板

```markdown
### 标题
[一句话准确描述]：<模块> - <行为> - <预期 vs 实际>

### 复现步骤
1. 精确到每一步，别人照做必复现
2. 截图 / 录屏 / 日志
3. 环境（设备 / OS / 版本 / 网络）

### 预期结果
<一句话>

### 实际结果
<一句话>

### 严重程度
Blocker | Critical | Major | Minor | Trivial

### 关联
- 相关 PR / 文档
- 重复 bug #
```

## 反模式

- ❌ 只跑 happy path
- ❌ 通过 = 没 bug
- ❌ 只写不能复现的 bug 报告
- ❌ 测试人员独立于开发完成后才接手（应全程参与）
- ❌ 回归测试全靠人工（会漏）
