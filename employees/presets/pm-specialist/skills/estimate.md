# 工时估算技能

## 核心公式
```
实际工时 = 乐观估时 × 复杂度系数 × (1 + 风险缓冲)
```

| 复杂度 | 系数 | 特征 |
|--------|------|------|
| S（琐碎）| 1.0 | CRUD、配置调整、单元测试 |
| M（常规）| 1.5 | 有现成模式，需适配；调试需花时间 |
| L（复杂）| 2.5 | 需设计、跨模块、有未知依赖 |
| XL（研究）| 4.0+ | 技术未验证、有 POC 风险 |

## 风险缓冲

- 依赖第三方 API / 库（未用过）：+30%
- 有遗留代码需改动：+20%
- 跨语言 / 跨平台：+25%
- 涉及性能 / 并发：+40%
- 多人协作（需评审 / 等待）：+15%

## 估算流程

### 1. 拆到足够小
> 单任务 ≤ 2 人天。超过就继续拆分。

### 2. 每个子任务独立估
不要整体估。先估"看起来最简单的子任务"，再估整体。

### 3. 记录估算依据
```markdown
### Task: 实现角色登录
- **乐观估时**: 0.5 天
- **复杂度**: M (需设计 Session 管理)
- **风险缓冲**: +30% (首次用 JWT)
- **最终估时**: 0.5 × 1.5 × 1.3 ≈ 1 天
- **依据**:
  - 后端已有注册接口，复用大部分代码
  - 前端需新增 Login 组件（参考现有 Register）
  - 未知：JWT 刷新机制是否需要
```

### 4. 三点估算（高风险任务）
- 乐观 (O)：一切顺利
- 最可能 (M)：常规进度
- 悲观 (P)：踩坑重做
- 期望值 = (O + 4M + P) / 6

## 反模式

- ❌ 拍脑袋"大概一天吧"
- ❌ 把"写代码时间"当成"交付时间"（忽略调试 / 评审 / 返工）
- ❌ 不给自己留缓冲（假设零 bug）
- ❌ 用"平均值"估高风险任务（应该用三点估算）
