# Sprint 规划技能

## 何时使用
- 收到"规划本周 / 下周 / 下个迭代"类指令时
- 项目启动或新里程碑开始时
- 团队成员需要明确"我这周该做什么"时

## 核心原则
**Sprint 计划必须是可执行的，不是愿景清单。**

- **具体**：每个任务都有明确的交付物（文件 / 原型 / 文档）
- **可度量**：有验收标准（功能演示 / 测试通过 / 用户验收）
- **有负责人**：不能是"团队负责"，必须到具体角色
- **有时限**：每个任务 ≤ 2 天，否则继续拆分
- **依赖显式**：A 任务依赖 B，必须写清楚

## 执行流程

### 步骤 1：读现状
```
必须先做：
- list_files 工作区根目录，找 README / ROADMAP / SPRINT-*.md
- read_file 读所有已存在的规划文档
- 搞清楚：项目阶段、当前已完成内容、未完成的阻塞点
```

### 步骤 2：产出计划文档
```markdown
# Sprint N — YYYY-MM-DD 到 YYYY-MM-DD

## 目标
- 1–3 条，对齐产品里程碑

## 任务清单

### Task 1: <标题>
- **负责人**: @<员工名>
- **优先级**: P0 | P1 | P2
- **估时**: X 天
- **依赖**: Task N (可选)
- **交付物**:
  - [ ] <具体文件 / 功能>
- **验收标准**:
  - <可验证的条件>
```

### 步骤 3：风险与假设
在文档末尾写出：
- 已做假设（若文档模糊）
- 已识别风险与缓解措施

## 反模式（禁止）

- ❌ "开发登录功能" —— 太模糊，拆成：UI 原型 / API 接口 / 数据库 Schema / 端到端测试
- ❌ "QA 团队负责测试" —— 必须指定具体角色（qa-tester）
- ❌ 估时 > 2 天的单任务（应继续拆分）
- ❌ 没有验收标准（怎么知道完成了？）
