# 范围检查技能

## 核心原则
**Scope creep（范围蔓延）是项目延期的第一大原因。每次新请求都必须先问"这在原计划里吗？"**

## 何时使用
- 用户提出新需求时
- 员工主动扩展任务时
- 代码 / 文档 / 功能膨胀速度超出计划时

## 检查清单

### 新需求来了，先问 4 个问题
1. **这在本次 Sprint 的目标里吗？**
   - 在 → 继续
   - 不在 → 走变更流程

2. **实现它需要多少工作量？**
   - < 1 小时 → 可以顺手做，但记录在"临时插入"清单
   - \> 1 天 → 必须走变更评审

3. **会影响哪些已计划的任务？**
   - 检查依赖图，标记受影响任务
   - 通知负责人

4. **不做的代价是什么？**
   - 可接受 → 推迟到下个 Sprint
   - 不可接受 → 走紧急变更流程

## 变更流程模板

```markdown
## 范围变更 #N
- **来源**: <用户 / XX 员工>
- **内容**: <一句话描述>
- **类型**: 新增 | 修改 | 删除
- **估时**: X 天
- **影响**:
  - 被挤出: Task #X (推迟到 Sprint N+1)
  - 需要调整: Task #Y (估时 +1 天)
- **决策**: 接受 | 拒绝 | 推迟
- **决策人**: <角色>
- **决策时间**: YYYY-MM-DD
```

## 反模式

- ❌ "顺手加个功能" —— 日积月累会让 Sprint 翻倍
- ❌ "这个简单的，半小时搞定" —— 低估 3-5 倍是常态
- ❌ 不告诉负责人就偷偷改他们的任务边界
