# 任务拆解技能

## 核心原则
**好的拆解让实施者一眼看懂"下一步要做什么"。**

拆解结果要满足：
- 每个子任务独立可完成（不等待外部输入）
- 每个子任务有明确完成标志（文件 / 测试 / 演示）
- 子任务之间的顺序清晰（哪些可并行，哪些有依赖）

## 拆解模板

### 规则 1：按交付物分组
```
大任务: 实现用户认证系统
  ├─ Task 1: 数据模型 (users 表 + 迁移脚本)
  ├─ Task 2: 注册接口 (POST /api/register)
  ├─ Task 3: 登录接口 (POST /api/login + JWT)
  ├─ Task 4: Session 中间件
  ├─ Task 5: 前端登录 UI
  ├─ Task 6: 前端注册 UI
  └─ Task 7: E2E 测试 + 文档
```

### 规则 2：每个任务都要有 DoD (Definition of Done)
```markdown
### Task 2: 注册接口
- **实现**: `api/auth/register.py`
- **输入**: `{email, password, name}`
- **输出**: `{user_id, created_at}` 或错误码
- **完成标志**:
  - [ ] Postman 能成功创建用户
  - [ ] email 重复返回 409
  - [ ] 密码 < 8 位返回 400
  - [ ] 密码以 bcrypt 哈希存储
  - [ ] 单元测试覆盖 3 种场景
  - [ ] OpenAPI 文档已更新
```

### 规则 3：拆到 ≤ 2 人天
超过就继续拆。典型的"太大"信号：
- 需要超过 200 行新代码
- 涉及超过 3 个文件
- 包含"以及"、"同时"、"并且"

## 并行度分析

拆完后画个简单依赖图：
```
T1 (数据模型)
  ├→ T2 (注册接口)  ──┐
  └→ T4 (中间件)    ──┤
                      ├→ T7 (E2E 测试)
T5 (前端注册) ────────┤
T6 (前端登录) ────────┘
T3 (登录接口) 依赖 T1, T4
```

标记 **可并行的任务**，方便安排多人或多 agent 并发执行。

## 反模式

- ❌ "实现完整登录系统"（太大）
- ❌ "写 Login.tsx 组件"（缺 DoD）
- ❌ 所有任务串行（没有识别并行机会）
- ❌ 漏掉测试任务（"改完代码就完事了"）
- ❌ 漏掉文档任务（未来会变技术债）
