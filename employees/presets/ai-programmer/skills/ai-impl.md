# AI 行为实现技能

## 核心选型

| 方案 | 适用 | 不适用 |
|------|------|--------|
| **FSM（状态机）** | 简单敌人、2–5 个状态 | 状态爆炸（>20 个转移） |
| **BT（行为树）** | 复杂行为、可视化调试 | 极度动态的决策 |
| **GOAP** | 需要"自动规划"的 NPC | 性能敏感 / 需要可预测性 |
| **效用 AI** | 需要权衡多目标（生存 / 攻击 / 资源） | 决策需要严格顺序 |

## 实施检查清单

### 设计阶段
- [ ] 列出所有 **行为原语**（Idle / Patrol / Chase / Attack / Flee）
- [ ] 画出 **转移条件**（什么情况下从 A 切到 B）
- [ ] 标记 **中断优先级**（被攻击时必须立即反应）
- [ ] 定义 **感知系统**（视野 / 听觉 / 距离）

### 编码阶段
- [ ] 每个行为独立成 class / 模块
- [ ] 转移条件独立成 **Condition** 对象（易测试）
- [ ] 提供 **Debug 可视化**（当前状态 / 黑板变量）
- [ ] 支持 **热重载**（不重启游戏调参）

### 测试阶段
- [ ] **单元测试**：单行为在隔离条件下的输出
- [ ] **场景测试**：多个 NPC 在典型战斗中的表现
- [ ] **极端测试**：100 个 NPC 同屏 / NPC 被快速驱离

## 性能要点

```
1 帧预算（60 FPS）：16.6 ms
AI 系统通常应 < 2 ms（12%）
```

- 不要每帧都跑完整决策。用 **tick 间隔**（如 0.2 秒）
- 距离玩家很远的 NPC，降频到 1 秒 / tick
- 感知计算用 **空间哈希**（不要 O(n²) 遍历）
- 路径查找用 **A\* + 路径平滑**，避免每帧重算

## 常见坑

- ⚠️ **状态泄漏**：切换状态时忘记重置计时器 / 目标
- ⚠️ **无限循环**：A 转 B、B 转 A 的条件都为真（加迟滞 / 最小停留时间）
- ⚠️ **决策延迟**：收到伤害后仍继续原行为（需要高优先级中断）
- ⚠️ **难以调试**：没有行为日志 / 可视化（先于功能实现）
