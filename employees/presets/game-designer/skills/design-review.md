# 设计评审技能

## 核心原则
**评审不是找茬，是"让设计变得更鲁棒"。每条意见必须可执行。**

## 评审框架（CIEA）

### C — Coherence（一致性）
- 新方案与现有模式是否冲突？
- 命名、风格、交互范式是否统一？
- 跨模块的边界是否清晰？

### I — Implementability（可实施性）
- 每个模块都有明确的实现路径吗？
- 有没有"魔法"假设（某个工具能做但未验证）？
- 需要的数据 / 资源 / 权限都准备好了吗？

### E — Edge Cases（边界情况）
- 空状态 / 首次使用 / 无网络 / 权限不足 时如何？
- 数据为 0 / 1 / 1000 / 1000000 时的表现？
- 并发 / 断线重连 / 崩溃恢复？

### A — Alternatives（可选方案）
- 还有没有别的方案？为什么选这个？
- 哪些决策是"不得不"，哪些是"可以再议"？

## 评审输出模板

```markdown
# 评审：<方案名> by <作者> @ YYYY-MM-DD

## 总体判断
✅ 通过 | ⚠️ 有条件通过 | ❌ 需重做

## 意见（按严重度排序）

### 🔴 阻断性（必须修改）
1. [问题] <问题描述>
   [影响] <如果不改会怎样>
   [建议] <可执行的改法>

### 🟡 重要（建议修改）
...

### 🟢 可选（风格 / 优化）
...

## 问题
对作者的 1–3 个关键问题。
```

## 反模式

- ❌ "这个不好，再改改" —— 无价值
- ❌ "你为什么不用 X？" —— 应该先理解作者思路再建议
- ❌ 评审语气居高临下 —— 对事不对人
- ❌ 只提问题不给方向 —— 至少给一个可行建议
- ❌ 所有问题都标"阻断性" —— 会让作者无所适从
