# 代码评审技能

## 核心原则
**评审不是让代码变完美，是让代码变得"足够好"并传递经验。**

## 审查维度（按优先级）

### P0 — 正确性
- 核心逻辑能正确处理主流程？
- 边界情况（空 / null / 0 / 最大值）？
- 错误处理：外部依赖失败时？
- 并发 / 竞态？

### P1 — 安全
- 用户输入是否转义 / 校验？
- SQL 注入 / XSS / CSRF？
- 敏感数据是否泄漏到日志 / 错误消息？
- 权限检查在正确的层（不能只在前端）？

### P2 — 可维护
- 命名是否自解释？（别人 6 个月后还看得懂吗？）
- 函数长度 < 50 行？职责单一？
- 复杂逻辑有注释解释"为什么"（而非"是什么"）？
- 有单元测试？覆盖主要分支？

### P3 — 性能
- 有 O(n²) 或更差的循环？
- 数据库查询：N+1？缺索引？
- 大对象是否按需加载？
- 资源是否正确释放（文件 / 连接 / 锁）？

### P4 — 风格
- 符合项目编码规范（缩进 / 命名 / 注释风格）？
- 无死代码 / 已注释代码？
- 依赖更新到最新稳定版？

## 评审格式

```markdown
## Review: <PR 标题>

### 🔴 必须修改（P0 / P1）
- [文件:行号] <问题> → <建议>

### 🟡 建议修改（P2 / P3）
...

### 🟢 可选（P4）
...

### ✅ 做得好的地方
<具体指出，帮助强化习惯>
```

## 反模式

- ❌ 所有问题都是"必须改"
- ❌ 只指出问题不给方案
- ❌ 挑剔个人风格偏好（应对项目规范）
- ❌ 批量找语法小问题却漏过架构问题
- ❌ 不夸一句好话（长期会打击积极性）
