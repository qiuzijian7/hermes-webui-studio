# 架构决策技能

## 核心原则
**每个"为什么选 X 不选 Y"的决策都值得记录。否则 3 个月后，包括你自己在内没人记得。**

## ADR 模板

```markdown
# ADR-<N>: <决策标题>

## 状态
Proposed | Accepted | Deprecated | Superseded by ADR-N

## 上下文
简述决策发生时的背景：
- 当时面临的问题
- 相关约束（时间 / 预算 / 团队能力 / 技术栈）

## 候选方案

### 方案 A: <名称>
- 优点: ...
- 缺点: ...
- 成本: ...

### 方案 B: <名称>
...

### 方案 C: <名称>
...

## 决策
我们选择 **<方案>**。

### 关键理由
1. ...
2. ...

## 后果

### 正面
- ...

### 负面（必须写）
- 需要接受的技术债
- 未来可能要付出的迁移成本

## 验证条件
在什么情况下需要重新评估这个决策？
- 当 <X> 超过 <阈值> 时
- 当团队规模达到 <N> 时
```

## 决策前的 4 个必问问题

1. **可逆性**：这个决策错了，多久能改回来？
   - 1 天 → 可大胆尝试
   - 1 周 → 需要多征求意见
   - 1 月+ → 必须慎重 + ADR

2. **影响面**：影响几个模块 / 多少人？

3. **假设**：我依赖了哪些"我以为成立"的前提？
   - 如："这个库会一直维护" → 怎么验证？

4. **最坏情况**：如果这个决策彻底失败，我们还有 Plan B 吗？

## 反模式

- ❌ 拍脑袋决定（无 ADR 记录）
- ❌ 只列方案不写"为什么不选"
- ❌ 省略"负面后果"章节
- ❌ 决策后从不回头评估
