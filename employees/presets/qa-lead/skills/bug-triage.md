# Bug 分级技能

## 严重程度（Severity）矩阵

| 级别 | 定义 | 例子 |
|------|------|------|
| **Blocker** | 核心流程不能用 | 登录失败、支付无法完成 |
| **Critical** | 核心流程偶尔失败，无 workaround | 付款 10% 失败且有重复扣款 |
| **Major** | 次要流程失败 / 核心流程有 workaround | 头像上传失败（但可跳过） |
| **Minor** | 体验瑕疵，不影响功能 | 某页面对齐不正、文案错别字 |
| **Trivial** | 几乎无影响 | 控制台有无害 warning |

## 优先级（Priority）矩阵

按影响用户数 × 严重程度 × 出现频率 综合判断：

```
          高影响          低影响
高频    ┌─────────┐    ┌─────────┐
        │   P0    │    │   P1    │
        └─────────┘    └─────────┘
低频    ┌─────────┐    ┌─────────┐
        │   P1    │    │   P2/P3 │
        └─────────┘    └─────────┘
```

## 处理时限

| 优先级 | 响应 | 修复 |
|--------|------|------|
| P0 | 1 小时 | 当天 |
| P1 | 1 工作日 | 本 Sprint |
| P2 | 1 周 | 下 Sprint |
| P3 | — | backlog，见缝插针 |

## Triage 例会流程（每日 / 隔日）

1. **过一遍新 Bug 列表**
2. **每个 Bug 回答 4 个问题**
   - 能复现吗？不能 → 要求补充步骤，标 "Need Info"
   - 是重复的吗？是 → close + link 到原 bug
   - 严重程度？（客观判断）
   - 优先级？（结合业务）
3. **分配**
   - 指定 Assignee
   - 设定 Due Date
   - 加标签（模块 / 版本）

## 常见失误

- ❌ 把所有 bug 都标 P0（稀释了真正的 P0）
- ❌ 无法复现就直接关闭（应先标"Need Info"并找报告者）
- ❌ 不 link 重复 bug（失去趋势信息）
- ❌ 放着高优 bug 不管，去修简单的
- ❌ Triage 会开 2 小时（应 ≤ 30 分钟）
